# pi_remote-windows-notify audit - 2026-07-09

## Round 1 - source/structure audit

Scope: `NotifyBridge.Common.ps1`, `notify-listener.ps1`, `pi-notify-broker.ps1`, `pi-notify-hotkey.ps1`, `pi-notify-popup.ps1`, `pi-notify-watchdog.ps1`, `pi-notify-refresh.ps1`, and `pi-notify-check.ps1`.

Findings:

- Listener -> broker popup delivery still used `HttpWebRequest`; prior runtime logs showed occasional broker POST timeout followed by legacy popup fallback even while the broker process was alive.
- Broker prewarm ran from a WinForms timer on the broker UI thread soon after popup show; full UIAutomation scans could take several seconds and compete with immediate click / hotkey activation.
- Broker prewarm queue had no same-target dedupe or closed-popup skip, so rapid replacement or quick activation could leave obsolete prewarm work behind.
- Broker background HTTP response writes treated client disconnects as generic `broker-bg-error`, creating noisy error-looking logs.

## Round 2 - runtime/log/UX audit

Runtime evidence before the fix showed:

- Config preserved `popupTimeoutSeconds=1800`, `popupHotkey=Alt+L`, `brokerEnabled=true`, and broker port `23119`.
- Listener and broker health endpoints returned OK.
- Listener log showed `broker-popup-sent ... timeout=1800`, but also a prior `broker-post-error` followed by `broker-post-failed fallback=popup-process`.
- Broker log showed prewarm scans taking multiple seconds, including examples around 3.4s to 5.7s.
- Broker log also showed `broker-bg-error` write exceptions after client disconnects.

Fixes applied:

- Replaced listener broker `/popup` POST with bounded loopback `TcpClient` + `BeginConnect` + explicit `Content-Length`, matching the hotkey fast path style.
- Added broker prewarm same-target dedupe, queue cap, recent-scan throttle, 1500ms delayed prewarm start, and closed-popup skip.
- Made broker background response write failures log as benign `broker-client-disconnect stage=response` instead of generic bg errors.

## Round 3 - regression/security/consistency audit

Checks performed:

- `git diff --check` passed.
- All `windows/*.ps1` files remained ASCII-decodable.
- `pi-notify-refresh.ps1 -SkipTunnel -SkipRemoteSync` passed on Windows.
- `pi-notify-check.ps1` passed on Windows, including syntax, ASCII, privacy, listener health, remote extension copies, token config modes, autostart VBS, tunnel singleton, hotkey worker count, and broker health.
- Runtime config still reports `popupTimeoutSeconds=1800` and `popupHotkey=Alt+L`.

Runtime evidence after the fix:

- Burst real `/notify` smoke returned `200 ok` three times; listener logged broker deliveries with `timeout=1800` and no broker fallback in that burst.
- Broker burst log showed only the final replacement popup displayed and prewarmed, reducing obsolete repeated work.
- Real `/notify` + Alt+L smoke logged `resident-fired hotkey=Alt+L`, `broker-activate-oldest-request`, `broker-activate-oldest`, `broker-activation-feedback`, target-cache hit, `broker-tab-selected cacheHit=True`, and feedback close.
- Quick activation before delayed prewarm logged `broker-prewarm-skip closed-popup`, proving closed popups no longer trigger obsolete prewarm scans.

## Remaining risk

Full UIAutomation tab scans can still take seconds on cache miss. The mitigation is to avoid starting prewarm immediately, dedupe/throttle repeated prewarm work, keep immediate activation feedback visible, and preserve fallback behavior. Further improvement would require a larger async STA scanning design and should be treated as a separate riskier refactor.
